note * all formats in this file (ex. RGBYYXX[OP]) are listed most significant byte to least significant byte. so in RGBXXYY[03], the most significant byte would be the R value.

internal gpu interface
    640x480, 24-bit color (maybe more modes later)

    op      desc

    0       invalid
    1       reserved [information query]
    2       reserved [set color mode]
    3       draw pixel (does not modify cursor pos and color)                                       XXYYRGB(0x03)
    4       set cursor color                                                                        ----RGB(0x04)
    5       set cursor position                                                                     XXYY---(0x05)
    6       draw text character (uses cursor position, modify cursor position for next character)   ----CC-(0x06)
    7       draw text character AT (modify cursor position for next character)                      XXYYCC-(0x07)

keyboard interface

    note * returns keycodes, not ascii or unicode values

    poll:       (0xF000000000000F)[POLL CODE]
    response:   [KEY7][KEY6][KEY5][KEY4][KEY3][KEY2][KEY1][RESPONSE CODE]       * keycodes will be returned lowest-first

    poll codes
        0   invalid, will result in an ERROR response
        1   poll keys actively pressed

    response codes
        0   error
        1   acknowledge


PORT BASED IO:

the general communication between the cpu and devices should be done through a port based device polling system,
allowing up to 224 ports for devices to be connected

communication protocol:
the cpu will poll the device, with a value on a 64 bit bus (we can use 8 bits and use a kind of bit-banging method)
the device will then recieve the poll and the value, allowing the device to recieve data from the cpu
the device can then poll the cpu through device specific IRQs, which are allocated and provided to the device
counting down from 0xFF.

example communication for a keyboard interface:

//aphelion asm below


pout ra, 0x3 // polls device 3 with the value stored in ra, with the value being output for at a minimum of 16 cycles
pin rb //places the value in the bus into rb

//keyboard handling code for reading values should be below

this system allows a call and response style communication from the cpu

in the reverse case, the device will poll the IRQ the cpu has told it has access to
the device should keep this value on for atleast 16 cycles, to allow the cpu to
start executing code

//device polls IRQ 0xF3
//code below that 0xF3 points to

pin rb //read the value the device has sent

//handling code

ret

this is not a bidirectional communication system, and is subject to bus clashes and race conditions. this means that
communication between devices and the cpu should be facilitated by some kind of bus arbiter, that allows only
one device to place values on the bus at a time. this is essentially a multiplexer, acting on >200 devices.
